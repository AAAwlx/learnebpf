## 正文

## 摘要

Linux 内核防火墙承担着包过滤、网络地址转换、连接跟踪和安全审计等任务。防火墙规则一旦配置不当，常见现象往往只表现为应用连接失败、服务不可达、转发中断或出口访问异常，传统日志又难以直接说明数据包在内核网络栈中被哪一个钩子点处理、是否被丢弃以及丢弃发生在收包、转发还是本地输出路径。针对这一问题，本文设计并实现了一种基于 eBPF 的 Linux 内核防火墙故障诊断系统。系统以 bpftrace 动态探针为内核观测层，在 ip_rcv、ip_forward、ip_local_out、ip_output、nf_hook_slow 等关键函数上采集源地址、目的地址、skb 指针、Hook 编号和 Netfilter 判决结果；以交互式包装脚本作为诊断控制层，支持按 IP、Hook、判决结果和 skb 地址进行组合过滤并生成日志；以 Python 分析器作为诊断分析层，对日志进行结构化解析，计算丢弃率、Hook 分布、函数调用频率、热点通信流和 skb 生命周期，将分散的跟踪输出转化为面向故障定位的诊断结论。系统围绕连接被拒、规则过严、本机输出受阻、转发路径异常、监控无输出和数据提取异常等故障建立诊断规则，使采集数据最终服务于故障判定、路径定位和配置复核。测试日志表明，当 192.168.18.156 访问 8.8.8.8 时，系统能够捕获 NF_INET_LOCAL_OUT 路径上的连续 DROP=-1 判决，并据此将故障定位到本机发包阶段的防火墙或本地输出规则。本文工作验证了 eBPF 动态跟踪在内核防火墙故障诊断中的可行性，为 Linux 网络故障排查提供了低侵入、可扩展的实现思路。

关键词：eBPF；Linux 防火墙；Netfilter；故障诊断；bpftrace

## Abstract

Linux kernel firewall mechanisms are widely used for packet filtering, network address translation, connection tracking and security auditing. When firewall rules are misconfigured, the symptom observed by users is usually vague, such as connection failure, service unreachability, forwarding interruption or abnormal outbound access. Conventional logs cannot always explain at which kernel hook a packet was processed, whether it was dropped, or whether the problem occurred on the receive path, forwarding path or local output path. To address this issue, this thesis designs and implements an eBPF-based diagnostic system for Linux kernel firewall failures. The system uses bpftrace probes as the kernel observation layer to collect source address, destination address, skb pointer, hook number and Netfilter verdict from key functions including ip_rcv, ip_forward, ip_local_out, ip_output and nf_hook_slow. An interactive wrapper script provides the diagnostic control layer, supporting combined filtering by IP address, hook number, verdict and skb address, as well as persistent logging. A Python analyzer parses trace logs, calculates drop ratio, hook distribution, function-call frequency, hot IP flows and skb lifecycle, and converts raw trace output into diagnostic conclusions. The system defines diagnosis rules for connection denial, overly strict rules, local output blocking, forwarding abnormality, no trace output and data extraction anomaly, ensuring that all extracted data is used for failure analysis and location. Test logs show that when 192.168.18.156 accesses 8.8.8.8, the system captures continuous DROP=-1 verdicts on NF_INET_LOCAL_OUT and locates the failure to the local output stage. The work verifies the feasibility of eBPF dynamic tracing in Linux firewall diagnosis and provides a low-intrusion and extensible approach for network troubleshooting.

Key words: eBPF; Linux firewall; Netfilter; fault diagnosis; bpftrace

## 第1章 绪论

### 1.1 研究背景

在 Linux 服务器、云主机和嵌入式网关通常通过 Netfilter 防火墙技术，通过iptables 或 nftables 等命令配置来实现访问控制。这里的防火墙主要针对于IP 层的内核协议栈，在进入到 IP 层之后会在路由前、本机输入、转发、本机输出和路由后等位置根据规则的设置对网络包进行处理。这样对于上层运行的进程来讲提供了便利性，做到了不用修改内核源码，就可以用轻量级的命令配置完成对于数据包路径的改变。但是由于这种实现对于上层屏蔽了所有的代码细节导致了故障发生时对于上层完全不可见难以对防火墙内部出现错误的地方进行定位有修改。业务侧看到的是连接超时或端口不可达，运维侧看到的是规则集、路由表和少量系统日志，二者之间缺少一条能够对应到内核处理路径的证据链。

在实际运维场景中，防火墙故障并不一定表现为明确的错误提示。一个访问外部地址失败的请求，可能是本机 OUTPUT 链丢弃，也可能是路由策略没有命中，还可能是上游网关阻断。一个外部用户无法访问本机服务的现象，既可能由 INPUT 链规则导致，也可能由服务未监听、DNAT 配置错误或安全组策略造成。传统排查方法通常依赖 `iptables -L -v -n`、`nft list ruleset`、`tcpdump`、`ping`、`curl` 等工具，这些工具能分别观察规则计数、链路包、应用连通性，却很难把单个数据包在内核中的处理轨迹串联起来。

Linux 内核网络栈的处理路径具有阶段性。接收方向的数据包需要经历网卡驱动、软中断、IP 层入口、路由判断、Netfilter Hook 和本地投递等过程；发送方向的数据包则由本机应用生成，经由本地输出 Hook、路由后处理和设备发送路径离开主机。防火墙故障诊断必须将“现象”翻译为“路径阶段”，再定位到对应的 Hook 和判决。本文课题正是基于这一思路，将 eBPF 动态跟踪能力用于 Netfilter 路径观测。这里可以让系统的使用者能够直观的观测到数据包在内核中所经历的处理过程以及处理结果。实现了内核数据转发的透明化。该系统还配备有日志分析工具可以按照用户设置的过滤规则从采集的日志当中精准定位问题，同时还接入了 AI 大模型对 eBPF 对采样数据进行分析。

### 1.2 研究意义

在该系统的故障诊断设计当中，研究的核心问题并不是简单统计网络包数量，而是面向三个问题进行设计：

1. 数据包是否进入内核网络栈
2. 经过了哪个 Netfilter Hook
3. 最终对网络包的处理结果是什么

eBPF 提供了在不修改内核、不重启系统的情况下动态观测内核函数的能力，适合用于这类低侵入排查任务。将 eBPF 动态跟踪与 Netfilter 路径知识结合，可以把原本分散在规则、日志和抓包结果中的信息组织成可解释的诊断链。

从工程价值看，基于 eBPF 的诊断方式可以降低故障定位对内核源码修改和系统重启的依赖。防火墙问题往往发生在正在运行的业务系统中，直接修改内核或替换防火墙模块成本较高，也会引入新的风险。eBPF 程序加载后附着在内核函数上，只在目标事件触发时执行观测逻辑，适合在临时排查、演示验证和线上辅助诊断中使用。本文采用 bpftrace 作为实现入口，虽然性能和工程化程度不如完整 libbpf 程序，但脚本形式能够清楚呈现探针位置、字段提取和输出内容，更适合毕业设计中对原理和实现过程的说明。

从教学与研究意义看，本文把 Netfilter 抽象概念落到可观测数据上。许多资料会介绍 PREROUTING、INPUT、FORWARD、OUTPUT 和 POSTROUTING 的含义，但学习者在实际系统中很难看到一个包如何经过这些位置。本系统通过 `hook_num`、`skb` 指针和判决结果建立对应关系，使理论中的 Hook 变成可分析的日志字段，也使防火墙规则的作用位置能够被验证。

### 1.3 研究内容

本文围绕 Linux 内核防火墙故障诊断系统展开设计与实现。研究内容包括 Netfilter 数据包处理路径分析、eBPF 动态探针采集方案设计、交互式过滤与日志模块实现、日志结构化解析与诊断规则构建、典型故障场景验证。在系统设计当中并不将采集到的数据日志或统计指标作为单独孤立的存在，而是实现了数据采集、分析与问题定位的逻辑闭环。在该系统当中，eBPF数据采集模块负责采集的源 IP、目的 IP、Hook 编号、skb 地址、判决结果分别服务于故障对象确认、故障阶段定位、单包生命周期复原和异常程度评估。

在系统实现层面该毕设项目主要实现了以下几个模块

1. eBPF数据采集模块，该模块主要负责在内核函数上打上探针采集前文中提到的各项采集指标，输出统一格式的路径记录
2. 用户交互模块，该模块为用户提供了一个交互式的页面，用户可以在这里选择抓包时的过滤模式以及在故障诊断时的分析模式
3. 故障诊断模块，该模块主要负责从多个纬度对采集到的数据进行故障诊断分析，同时还引入了 AI agnet 智能诊断的功能

以这三个程序分别对应采集、控制和分析三个环节，实现了对项目的解耦，提升了系统的可扩展性。

针对故障诊断，在诊断模型层面，本文将常见故障分为本机输出受阻、入站服务不可达、转发路径异常、规则过严或误封、监控无输出、数据提取异常六类。每类故障都定义了对应的观测点和判断依据。例如本机输出受阻主要观察 `NF_INET_LOCAL_OUT` 与 `DROP` 判决，入站服务不可达主要观察 `PRE_ROUTING` 与 `LOCAL_IN`，转发路径异常则需要关注 `FORWARD` 和后续输出路径是否连续。这样处理后，采集到的每个字段都能进入故障判断，而不是停留在“监控到了很多日志”的层面。

## 第2章 相关技术与需求分析

### 2.1 eBPF 与动态跟踪技术

eBPF 是 Linux 内核提供的可编程执行机制，程序加载前会经过内核侧静态检查，避免对运行系统造成破坏。bpf 系统调用文档指出，eBPF 能够访问辅助函数和共享数据结构，内核在加载前会分析程序安全性。对于故障诊断任务而言，eBPF 的价值在于提供了一种简单的编程范式，可以用轻量级的程序对内核级别的细粒度问题进行非侵入式的观测。让以往暗藏于内核之中难以被捕捉到的错误数据在用户态得以观测。

eBPF 程序运行在内核预定义的事件上下文中，常见挂载点包括跟踪点、Kprobes、网络收发路径、套接字过滤和 XDP 等。程序加载时，内核验证器会检查指令边界、内存访问、循环形式、指针使用和辅助函数调用，防止不安全程序破坏内核状态。通过 map，eBPF 程序能够在事件之间保存状态，也能够与用户态程序交换数据。本课题使用 bpftrace 时，底层仍然依赖 eBPF 机制，只是由 bpftrace 负责完成脚本解析、程序生成、加载和输出。

防火墙诊断对 eBPF 的需求主要体现在动态性和上下文性。动态性是指诊断程序可以在故障发生时临时加载，完成排查后卸载，不需要长期修改内核。上下文性是指探针可以读取当前内核函数参数、返回值以及相关结构体字段，直接获得 `struct sk_buff`、`struct nf_hook_state` 等与数据包处理密切相关的信息。传统抓包工具能看到链路层或网络层报文，但无法直接看到内核防火墙返回的判决；eBPF 探针则可以补足这一缺口。

### 2.2 Kprobes 与 bpftrace

Kprobes 支持在内核函数入口或返回点动态插入探针，用于非侵入地收集调试和性能信息。bpftrace 在此基础上提供脚本化语言，支持 kprobe 和 kretprobe，能够快速绑定内核函数并读取参数、返回值和内核结构体字段。本系统采用 bpftrace 的原因在于毕业设计场景更重视可解释性、可演示性和快速迭代，脚本能够清晰呈现从 skb 提取到输出格式化的完整过程。

Kprobe 附着在函数入口处，适合读取函数参数和调用时的上下文；Kretprobe 附着在函数返回处，适合读取返回值。Netfilter 的关键函数 `nf_hook_slow` 在入口处能够拿到 `skb` 和 Hook 状态，在返回处才能知道最终判决。本文在入口探针中用线程 ID 暂存 `skb` 指针，在返回探针中读取 `retval` 并恢复对应 `skb`，由此把“进入哪个 Hook”和“返回什么判决”关联到同一条输出链路中。

bpftrace 脚本中的 `arg0`、`arg1`、`arg2` 表示内核函数参数，`retval` 表示返回值。项目中的 IP 层函数参数位置并不完全一致，例如 `ip_rcv` 的 `skb` 位于第一个参数，而 `ip_local_out`、`ip_output` 等函数中的 `skb` 位于后续参数，因此采集脚本必须按函数原型分别处理。这个细节对诊断准确性有直接影响：如果参数位置错误，系统可能读取到无效地址，进而输出错误 IP 或无法输出记录。

### 2.3 Netfilter 防火墙机制

Netfilter 是 Linux 内核中的包过滤框架，允许内核模块在网络栈不同位置注册回调函数。官方项目说明中列出的能力包括包过滤、NAT、日志、用户态队列和包修改。IPv4 路径中的五个典型 Hook 分别对应 PREROUTING、LOCAL_IN、FORWARD、LOCAL_OUT 和 POSTROUTING。防火墙规则实际生效时，数据包会在这些 Hook 上进入规则处理逻辑，返回 ACCEPT、DROP、QUEUE、STOLEN 等判决。

Netfilter 与 iptables 的关系可以理解为内核框架与规则管理工具之间的关系。Netfilter 在内核网络路径中提供 Hook 点，iptables 或 nftables 则向这些 Hook 对应的链中组织规则。filter 表通常负责包过滤，nat 表负责地址转换，mangle 表用于修改报文头或标记，raw 表常用于连接跟踪前的特殊处理，security 表与安全标记有关。不同表注册到不同 Hook 上，并按照优先级被调用。

当数据包进入某个 Hook 后，内核会按照注册顺序遍历处理函数。以 iptables 为例，最终会进入 `ipt_do_table` 完成规则匹配。若规则返回 ACCEPT，数据包继续后续处理；若返回 DROP，内核释放对应 `skb` 并终止处理；若返回 QUEUE，数据包会交由用户态队列处理；若返回 STOLEN，数据包由某个模块接管。对故障诊断来说，最关键的是区分“没有到达 Hook”“到达 Hook 后被 DROP”“到达 Hook 并 ACCEPT 但后续路径异常”这三种情况。

IPv4 收包方向中，`ip_rcv` 是 IP 层入口，随后通过 `NF_HOOK` 触发 `NF_INET_PRE_ROUTING`。路由判断后，目标为本机的包进入 `ip_local_deliver` 并触发 `LOCAL_IN`，需要转发的包进入 `ip_forward` 并触发 `FORWARD`。本机发出的包会经过 `LOCAL_OUT`，路由后准备发送的包会经过 `POST_ROUTING`。本文选择这些路径函数作为探测点，是为了让系统能把防火墙判决放回完整 IP 层路径中解释。

### 2.4 防火墙故障类型

结合项目实现和 Netfilter 处理路径，本文将系统需要诊断的故障分为六类。其一是本机输出受阻，表现为本机访问外部地址失败，诊断特征是 NF_INET_LOCAL_OUT 上出现 DROP 或负数返回。其二是入站服务不可达，表现为外部主机访问本机服务失败，诊断特征是 PRE_ROUTING 或 LOCAL_IN 路径上目标地址相关数据包被丢弃。其三是转发链故障，表现为网关或路由节点无法转发流量，诊断特征是 FORWARD Hook 命中后出现 DROP 或缺少后续输出路径。其四是规则过严或误封，表现为特定源、目的地址持续被拒绝，诊断特征是同一 IP 对或同一 skb 在短时间内重复出现丢弃判决。其五是监控无输出，可能由没有触发流量、过滤条件过窄、权限不足或 bpftrace 未安装引起。其六是数据提取异常，例如在某些 Hook 中 skb->data 并不指向预期 IP 头，输出可能出现 0.0.0.0 或字段缺失，需要通过更稳妥的 network_header 方式改进。

本机输出受阻类故障通常由 OUTPUT 链规则、策略路由配合标记规则、主机安全软件或容器网络策略引起。它的典型特征是应用位于本机，目标位于外部或其他网段，数据包已经由本机协议栈生成，但尚未离开主机就被拦截。系统观察到 `hook_num:3` 与 DROP 集中出现时，应优先检查本机 OUTPUT 链、nftables output chain、按 UID/进程/目的地址匹配的规则，以及是否存在默认 DROP 策略。

入站服务不可达类故障需要区分“包未到达主机”和“到达后被防火墙拒绝”。若系统在 `PRE_ROUTING` 上看不到目标为本机的数据包，则说明故障可能位于上游链路、交换机、云安全组或抓取条件。若 `PRE_ROUTING` 可见而 `LOCAL_IN` 返回 DROP，则说明数据包已进入主机并经过路由判断，问题更可能位于 INPUT 链或本机服务相关安全策略。这个区分能够避免把网络链路问题误判为本机防火墙问题。

转发路径异常类故障常见于 Linux 网关、容器宿主机、虚拟化节点和软路由。诊断时不能只看源和目的地址，还要看包是否经过 `FORWARD`，是否在转发后进入 `POST_ROUTING`，以及是否出现 NAT 相关变化。若 `FORWARD` 上出现 DROP，说明规则拒绝转发；若没有 DROP 但缺少后续输出路径，则可能是路由项、反向路径过滤、出接口状态或 NAT 配置导致。

监控无输出与数据提取异常虽然不是防火墙规则故障，但会影响诊断系统本身的可信度。本文将它们纳入故障类型，是因为诊断工具必须能解释“为什么没有证据”。没有输出时，系统应检查权限、依赖、内核符号、过滤条件和流量触发；字段异常时，系统应提示采集逻辑风险，而不是继续给出过度确定的网络故障结论。

### 2.5 系统需求

系统功能需求包括内核路径采集、交互式过滤、日志持久化、本地统计分析和可选智能分析。诊断需求要求每条采集数据都能映射到诊断含义：IP 地址用于确定故障对象，Hook 编号用于定位故障阶段，判决结果用于判断是否被防火墙拒绝，skb 地址用于复原单个数据包生命周期，函数调用频率用于发现路径集中度，日志时间和记录数量用于判断异常是否持续。非功能需求包括低侵入、可复现、易部署和便于答辩演示。

内核路径采集需求要求系统覆盖接收、本地投递、转发、本地输出和最终输出等关键路径。只采集 `nf_hook_slow` 可以看到 Hook 和判决，但无法完整说明数据包进入 IP 层前后的路径；只采集 IP 层函数可以看到路径，却无法说明是否被 Netfilter 规则丢弃。本文将两类探针结合起来，使系统既能追踪路径，也能解释判决。

交互式过滤需求来自实际排查习惯。故障通常不是“所有网络都有问题”，而是某个目标地址不可达、某个源地址被拒绝或某条链上的规则异常。系统需要允许用户按 IP、Hook、结果和 skb 地址逐步收窄范围。日志持久化需求则用于复盘和论文测试，避免只依赖终端瞬时输出。分析需求要求系统输出可读报告，并将统计数字转化为诊断语言，例如“DROP 集中在 LOCAL_OUT”比“DROP 数量较多”更有定位价值。

非功能需求中，低侵入意味着系统不修改防火墙规则，不改变网络包处理逻辑，只观察并记录；可复现意味着同一日志文件能够被反复分析并得到一致结论；易部署意味着在具备 bpftrace 和 root 权限的 Linux 环境中即可运行；可演示意味着系统操作流程清晰，适合通过故障注入或已有日志在答辩中展示。

## 第3章 系统总体架构设计

### 3.1 架构设计原则

系统采用分层架构，将内核观测、用户交互、数据存储和诊断分析解耦。内核观测层只负责在关键函数上提取必要字段，避免在探针中写入复杂判断；交互控制层负责将用户的故障假设转化为过滤条件；数据存储层保留原始证据，便于复核；诊断分析层再把原始记录聚合为面向故障的结论。这种划分能够保证系统既适合实时排查，也能支持事后分析。

架构设计遵循“采集尽量客观、分析尽量解释”的原则。内核采集层只输出事实字段，包括函数名、地址、skb、Hook 和返回值，不在内核探针中判断规则是否合理。用户态分析层再根据故障模型判断异常原因，这样可以降低探针复杂度，也便于后续替换分析策略。若未来接入 Web 界面或数据库，内核采集层仍可保持稳定。

系统还遵循“从故障现象反推观测点”的原则。用户报告本机访问失败时，诊断重点放在 LOCAL_OUT；用户报告外部访问失败时，诊断重点放在 PRE_ROUTING 和 LOCAL_IN；用户报告网关转发失败时，诊断重点放在 FORWARD 和 POST_ROUTING。交互控制层的过滤菜单正是按照这种思路设计，使使用者不必一次处理全量内核输出。

### 3.2 总体架构

在故障诊断的系统总体架构由四个主要模块构成。内核采集模块由 netfilter_trace.bt 实现，探测 ip_rcv、ip_rcv_finish、ip_local_deliver、ip_forward、ip_local_out、ip_output、ip_finish_output 和 nf_hook_slow 等函数。交互控制模块由 netfilter_wrapper.sh 实现，提供 IP、Hook、结果和 skb 四类过滤条件。日志管理模块以 logs 目录中的时间戳文件保存监控结果。诊断分析模块由 netfilter_analyzer.py 实现，将日志行解析为 NetflowRecord，并在 NetworkAnalyzer 中维护统计状态。

系统数据流可以概括为“内核事件触发、探针输出记录、包装层过滤、日志落盘、分析器聚合、诊断报告生成”。当网络包进入 IP 层或 Netfilter 函数时，bpftrace 探针被触发并输出一行文本。包装脚本根据用户配置决定是否对输出进行过滤，并在日志模式下写入文件。分析器读取标准输入或日志文件，将每行文本转换为 `NetflowRecord`，再更新 `NetworkAnalyzer` 中的计数器和映射表。最终报告中的每一项指标都来自这些结构化状态。

内核采集模块与分析模块之间采用文本日志作为接口。这样设计的优点是实现简单、可读性强、便于保存和展示；不足是日志解析依赖输出格式，字段扩展时需要同步修改正则表达式。对于毕业设计阶段，文本日志能清楚展示系统工作过程；对于生产系统，后续可改为 ring buffer 或 perf event 输出结构化二进制事件。

系统逻辑架构可以用四层表示：采集层面向内核网络路径，控制层面向用户诊断意图，存储层面向证据留存，分析层面向故障结论。采集层产生的字段是原始事实，控制层保证事实与故障对象相关，存储层保证事实可复核，分析层把事实解释为路径阶段和可能原因。四层缺一不可，单独采集而无分析会造成信息堆积，单独分析而无内核证据则会变成经验推断。

### 3.3 诊断数据模型

本文将诊断记录抽象为六元组：函数名、源地址、目的地址、skb 地址、Hook 编号和判决结果。函数名说明数据包所处的内核路径，源地址和目的地址确定通信对象，skb 地址关联同一数据包在多个函数中的轨迹，Hook 编号对应防火墙规则链位置，判决结果给出是否放行的直接证据。分析器在此基础上维护总记录数、ACCEPT/DROP 数、唯一源和目的地址集合、Hook 分布、函数调用频率、IP 对频率以及 skb 跟踪表。

函数名字段用于还原路径阶段。例如 `ip_rcv` 表示数据包已经进入 IPv4 接收路径，`ip_forward` 表示路由判断后需要转发，`ip_local_out` 表示数据包由本机生成，`ip_finish_output` 表示即将完成输出。Hook 编号字段用于映射 Netfilter 规则链，0 到 4 分别对应 PRE_ROUTING、LOCAL_IN、FORWARD、LOCAL_OUT 和 POST_ROUTING。判决字段用于识别处理结果，ACCEPT 表示继续处理，DROP 或负返回值表示包被拒绝或处理失败。

skb 地址字段在诊断模型中承担关联键的作用。Linux 内核使用 `struct sk_buff` 表示网络包，系统输出 skb 指针后，可以观察同一个 skb 是否连续经过多个函数。若一个 skb 只出现于 `nf_hook_slow` 和 DROP 记录，说明其生命周期在该 Hook 附近终止；若同一 skb 依次出现在接收、路由、转发和输出函数中，则说明路径较完整。虽然 skb 地址会被内核复用，不能跨长时间窗口随意关联，但在短时诊断窗口内具有较高实用价值。

分析器维护的统计状态服务于不同层次的诊断。总记录数反映样本规模，唯一源和目的地址反映故障范围，IP 对频率反映异常是否集中，Hook 分布反映异常阶段，函数调用频率反映采集路径，skb 跟踪表反映单包生命周期。只有这些指标组合使用，系统才能从“看到 DROP”进一步推导出“哪个对象在什么阶段被持续拒绝”。

### 3.4 故障诊断流程

诊断流程从故障现象开始，而不是从工具输出开始。用户先根据现象选择过滤策略：连接某个外部地址失败时按目标 IP 与 DROP 过滤，入站服务不可达时按本机服务地址或 LOCAL_IN 过滤，怀疑转发故障时按 FORWARD Hook 过滤，怀疑单包路径异常时按 skb 地址复查。系统采集数据后，分析器计算异常指标并给出路径判断。若 DROP 集中在 LOCAL_OUT，诊断结论指向本机输出规则；若 DROP 集中在 LOCAL_IN，结论指向本机入站防火墙；若 FORWARD 命中后没有 POST_ROUTING 或输出函数，结论指向转发链或路由策略；若没有任何输出，则转入环境和过滤条件诊断。

一次完整诊断通常分为四个阶段。现象确认阶段明确故障对象，例如源主机、目标地址、服务端口和操作方向。观测配置阶段选择过滤条件，例如按目标 IP、按 Hook 或按 DROP 结果过滤。证据采集阶段运行 bpftrace 探针并触发故障流量，保证日志中包含与故障相关的数据。分析判断阶段读取日志，检查判决结果、Hook 分布和 skb 生命周期，形成故障结论与后续检查建议。

诊断流程也需要处理证据不足的情况。若日志记录数少于分析阈值，系统不应给出强结论，而应提示触发流量不足或过滤条件过窄。若仅有 IP 层函数记录而没有 `nf_hook_slow` 记录，则可能是系统没有命中 Netfilter 钩子、规则链未注册或探针未成功触发。若仅有 `nf_hook_slow` 而缺少 IP 层路径记录，则应检查对应函数是否在当前内核版本中被内联、重命名或路径未覆盖。

## 第4章 关键模块设计与实现

### 4.1 内核采集模块实现

netfilter_trace.bt 通过多个 kprobe 绑定 IPv4 路径函数。在 ip_rcv、ip_local_deliver、ip_forward 等入口处，脚本从函数参数中取得 struct sk_buff 指针，再读取 skb->data 并转换为 struct iphdr，从中提取源地址和目的地址。对 nf_hook_slow，脚本在函数入口读取 nf_hook_state 中的 hook 字段，并把当前线程的 skb 保存到 bpftrace map；在 kretprobe:nf_hook_slow 中读取返回值，将返回值 1 识别为 ACCEPT，将 -1 识别为 DROP，并输出对应 skb。入口和返回探针的配合使系统既能知道数据包经过哪个 Hook，也能知道处理结果。

采集脚本对不同内核函数分别处理 `skb` 参数位置。`ip_rcv`、`ip_local_deliver` 和 `ip_forward` 的 `skb` 位于 `arg0`，而 `ip_rcv_finish`、`ip_local_deliver_finish`、`ip_local_out`、`ip_output`、`ip_mc_output` 和 `ip_finish_output` 中的 `skb` 位于后续参数。脚本在每个探针中都执行四个步骤：取得 `skb` 指针，读取 IP 头位置，将 IP 头转换为 `struct iphdr`，输出源地址、目的地址和 skb 地址。重复这些步骤看似冗余，但能让每个探针逻辑独立，便于定位某个函数参数或字段读取错误。

`nf_hook_slow` 是系统采集中的关键位置。该函数位于 Netfilter 慢路径处理过程中，能够反映某个 Hook 上规则处理后的最终返回值。入口探针读取 `struct nf_hook_state` 中的 `hook` 字段，得到当前 Hook 编号；返回探针读取 `retval`，将 `1` 解释为继续执行后续函数，将 `-1` 等负值解释为丢弃或错误返回。项目日志中出现的 `[DROP=-1]` 正是这一路径产生的结果，它比单纯抓包更接近防火墙判决本身。

输出格式统一为方括号标识函数或判决，随后给出源地址、目的地址和 skb 地址。统一格式降低了后续解析复杂度，也便于用户在终端中直接阅读。对于 Hook 记录，输出中额外加入 `hook_num` 字段，使用户可以直接判断数据包处于 PRE_ROUTING、LOCAL_IN、FORWARD、LOCAL_OUT 还是 POST_ROUTING。

| kprobe:ip_output | src_ip、dst_ip、skb | IP 层输出阶段 | 观察路由后输出路径 |
| kprobe:nf_hook_slow | src_ip、dst_ip、skb、hook_num | Netfilter Hook 处理入口 | 定位具体防火墙 Hook |
| kretprobe:nf_hook_slow | verdict、skb | Netfilter Hook 返回点 | 判断 ACCEPT、DROP 或其他结果 |

### 4.2 交互控制模块实现

netfilter_wrapper.sh 提供菜单式控制界面。用户可以设置 FILTER_IP、FILTER_HOOK、FILTER_RESULT 和 FILTER_SKB，generate_filter 函数将这些条件转换为 grep 管道。多个条件之间采用 AND 关系，这符合故障排查时逐步收窄范围的习惯。例如，当用户怀疑 192.168.18.156 访问 8.8.8.8 被拒绝时，可以同时设置目标 IP 和 DROP 结果，只保留与该故障直接相关的证据。

包装脚本的菜单围绕诊断动作组织，而不是围绕底层命令参数组织。IP 过滤用于确定故障对象，Hook 过滤用于聚焦故障阶段，结果过滤用于快速查看 ACCEPT、DROP 或其他返回，skb 过滤用于复查单个数据包生命周期。用户可以先不设置过滤条件运行全量监控，从输出中找到可疑 skb 或目标地址，再返回菜单逐步增加过滤条件。

`generate_filter` 函数将用户配置转换为管道过滤命令。若未启用过滤，函数返回 `cat`，系统输出全量记录；若设置 IP、Hook、结果或 skb 条件，函数会追加对应 `grep` 表达式。多个表达式串联后形成 AND 关系，只有同时满足所有条件的记录才会显示。该设计实现简单，也便于用户理解：每增加一个条件，输出范围就更接近当前故障。

脚本在启动监控前检查 `bpftrace` 和脚本文件是否存在，在 AI 分析模式中还检查 `python3` 和分析器脚本是否存在。这些检查本身也是诊断系统可靠性的一部分。若依赖缺失，脚本直接给出错误提示，避免用户误以为“没有输出就是没有网络包”。

### 4.3 日志管理模块实现

日志文件以 netfilter_trace_YYYYMMDD_HHMMSS.log 命名，存放在 logs 目录。日志模式通过 tee 同时向屏幕和文件写入记录。该设计使实时排查和事后复盘可以共用同一份证据。对于毕业设计演示场景，日志还能作为固定输入，避免现场网络状态变化影响诊断结果复现。

日志命名使用时间戳可以区分不同诊断会话。一次故障排查往往需要多轮尝试，例如先全量观察，再按 IP 过滤，再按 DROP 过滤。独立日志文件能够保留每轮观察结果，便于比较过滤前后的差异。论文中的测试分析就使用项目目录下已有日志作为固定输入，避免重新触发网络环境带来的不确定性。

日志内容采用与终端输出一致的格式，便于人工阅读和程序解析。人工复查时，可以直接搜索 `DROP`、目标 IP 或 `hook_num`；程序分析时，`NetflowRecord` 使用正则表达式提取结构化字段。日志没有采用复杂数据库，是因为本课题的重点在于内核路径诊断和故障规则说明，文本文件已经能够支撑原型系统验证。

### 4.4 诊断分析模块实现

netfilter_analyzer.py 中的 NetflowRecord 负责解析单行日志，通过正则表达式提取函数名、IP 地址、skb 地址、Hook 编号和判决结果。NetworkAnalyzer 则维护累积统计并生成本地报告。报告中的总体统计用于判断故障规模，Hook 分布用于定位故障阶段，函数调用 TOP5 用于判断采集路径是否集中，热门 IP 流用于识别故障对象，skb 生命周期用于分析同一数据包是否完成了预期路径。

`NetflowRecord` 的解析过程对应采集字段的数据模型。函数名通过方括号内容提取，IP 地址通过 `源地址 -> 目的地址` 形式提取，skb 地址兼容 `skb:` 与 `skb=` 两种输出形式，Hook 编号通过 `hook_num` 提取。解析成功后，记录被加入分析器；若记录缺少函数名或 IP 地址，系统会认为该行不完整而不参与统计。这样的处理可以减少无效输出对诊断结果的干扰。

`NetworkAnalyzer` 使用 `Counter` 和集合维护统计状态。`hook_distribution` 统计不同 Hook 的命中次数，`function_calls` 统计函数记录分布，`ip_pairs` 统计源目的地址对，`skb_tracking` 以 skb 地址为键保存同一数据包的相关记录。报告生成时，分析器根据 ACCEPT 与 DROP 数计算丢弃比例，根据 Hook 分布判断阶段，根据 skb 记录数量估计平均处理步数。

现有分析器对 `[DROP]` 的识别较直接，对 `[DROP=-1]` 这类负返回格式在统计上会被当作函数名处理。论文在测试分析中手工结合日志内容将其视为丢弃判决，这也暴露了实现中需要改进的地方。后续可在 `NetflowRecord.parse` 中增加 `DROP=` 模式识别，把负返回统一归一化为 DROP，并保存原始错误码字段，以便区分普通丢弃和带错误码的拒绝。

分析器生成报告时采用从总体到细节的顺序。总体统计说明记录数量、接受数量、丢弃数量和唯一地址数量；Hook 分布说明故障集中在哪个防火墙阶段；函数调用频率说明当前日志主要来自哪类内核路径；热门 IP 流说明故障对象是否集中；安全分析根据丢弃比例和公网/内网来源给出风险提示；skb 生命周期分析根据同一 skb 的记录数量判断路径是否完整。这个顺序符合故障排查习惯，先判断范围，再定位阶段，随后查看对象和单包轨迹。

表4-2 分析指标与诊断含义

|---|---|---|
| 总记录数 | 所有有效 NetflowRecord | 判断样本是否足以支撑结论 |
| ACCEPT/DROP 数 | verdict 字段 | 判断流量是被放行还是被拒绝 |

### 4.5 智能分析扩展

系统预留 AI 深度分析模式。当配置 Anthropic API Key 后，分析器会把统计信息、Hook 分布、热门函数、热门 IP 流和最近记录整理为上下文，交由大语言模型生成安全风险、性能瓶颈和优化建议。本文将其作为扩展能力处理，核心诊断结论仍以本地结构化统计为基础，从而保证在无网络或未配置 API 的环境下系统仍可运行。

AI 分析模式在系统中的定位是辅助解释，而不是替代本地诊断规则。分析器会先生成本地统计报告，再将统计信息、最近记录和热点流量提供给模型。模型适合根据上下文给出排查建议，例如检查规则顺序、默认策略或异常源地址；但最终结论仍应以本地日志字段为依据。这样可以避免外部模型在证据不足时产生过度推断。

在毕业设计实现中，AI 模式还体现了系统可扩展性。内核采集层和日志层无需修改，只要分析层能够提供结构化上下文，就可以接入不同的智能分析后端。若未来部署在无外网环境中，也可以完全关闭 AI 模式，仅使用本地分析报告完成故障诊断。

### 4.6 关键实现限制

当前 bpftrace 版本直接使用 skb->data 读取 IP 头，在多数 IPv4 路径上便于理解和演示，但在部分 Hook 或特殊包处理阶段可能出现数据指针不符合预期的问题。分析器对 [DROP] 格式的判决识别较直接，而测试日志中出现的 [DROP=-1] 属于负返回值形式，论文分析时将其归入丢弃判决。后续实现可在解析阶段增加判决归一化，并在内核采集阶段使用 skb->head 与 network_header 组合提高字段提取稳健性。

字段提取限制主要来自内核网络包表示的复杂性。`skb->data` 在不同处理阶段可能指向不同协议层位置，尤其在经过封装、分片、GRO/GSO 或某些 Hook 调整后，直接把 `skb->data` 转换为 IP 头存在风险。更稳妥的方式是读取 `skb->head + skb->network_header`，并在读取前检查 network header 是否有效。bpftrace 对复杂边界检查和类型适配的表达能力有限，这也是后续迁移到 libbpf 的原因之一。

探针稳定性也与内核版本有关。不同内核版本中函数名、参数位置、内联情况可能发生变化，某些函数在发行版内核中可能被优化或不可见。当前系统适合在实验环境和指定内核版本中验证诊断思路，若用于更广泛部署，需要增加内核版本检测、符号存在性检查和多套探针适配方案。

## 第5章 故障诊断场景与测试分析

### 5.1 测试环境与数据来源

测试材料来自项目 logs 目录下的 netfilter_trace_20260427_232905.log。该日志共 58 行，通信对象均为 192.168.18.156 到 8.8.8.8，其中包含 30 条 hook_num:3 记录和 29 条 DROP 相关记录。Hook 3 对应 NF_INET_LOCAL_OUT，说明数据包在本机发出阶段进入 Netfilter 处理。这个日志适合作为本机输出受阻故障的样例。

测试日志的记录形式具有明显规律：一类记录为 `[nf_hook_slow] 192.168.18.156 -> 8.8.8.8 | skb:... | hook_num:3`，另一类记录为 `[DROP=-1] 192.168.18.156 -> 8.8.8.8 | skb=...`。两类记录交替出现，说明同一故障流量反复进入本机输出 Hook，并在返回阶段得到负返回值。日志中没有 ACCEPT 记录，也没有其他源目的地址对，故障对象和故障阶段都比较集中。

本文对该日志进行了三类统计。记录数量统计用于确认样本规模，`hook_num:3` 统计用于确认故障阶段，IP 对统计用于确认故障对象。结果显示，58 行日志全部指向 `192.168.18.156 -> 8.8.8.8`，30 行包含 `hook_num:3`，29 行包含 DROP 相关内容。由于 Hook 3 对应本机输出路径，故障结论应优先指向本机 OUTPUT 链或本地输出相关策略。

### 5.2 本机输出受阻诊断

当用户报告本机无法访问外部 DNS 或公网地址时，传统排查通常从 ping、curl、路由表和防火墙规则开始。本系统的诊断路径更靠近内核处理过程。日志显示，同一源目的地址对在 NF_INET_LOCAL_OUT 上反复进入 nf_hook_slow，并伴随 DROP=-1 返回。由于 LOCAL_OUT 发生在本机生成的数据包进入协议栈后、真正发出之前，因此诊断结论不是外部网络不可达，也不是对端拒绝，而是本机输出路径上的防火墙规则或本地安全策略阻止了该流量。

该诊断结论来自三个证据链。其一，源地址为本机内网地址 `192.168.18.156`，目的地址为公网 DNS 地址 `8.8.8.8`，符合本机访问外部地址的方向。其二，Hook 编号为 3，对应 `NF_INET_LOCAL_OUT`，说明数据包已经由本机产生并进入本地输出防火墙处理，而不是在入站或转发路径上被处理。其三，返回记录为 `DROP=-1`，说明 Netfilter 处理没有放行该包。三个证据组合后，可以排除“对端服务未响应”作为首要原因，因为数据包在离开本机前已经被处理失败。

在实际排查中，根据该结论应继续检查 `iptables -S OUTPUT`、`iptables -L OUTPUT -v -n` 或 `nft list chain` 中 output 链的规则，重点关注目的地址、协议、端口、连接状态和默认策略。若系统使用容器或虚拟化网络，还应检查容器运行时生成的规则、网络命名空间中的规则以及主机安全软件策略。本文系统不直接修改这些规则，而是提供定位依据，帮助使用者减少盲目排查范围。

### 5.3 规则过严或误封诊断

规则过严通常表现为某个源地址、目的地址或端口的流量持续被拒绝。系统通过 IP 对频率和判决结果联合判断这一类故障。若热门 IP 流 TOP5 中某一对地址占比接近全部记录，同时 DROP 数量高于 ACCEPT 数量，就说明故障不是偶发丢包，而是规则稳定命中。测试日志中 192.168.18.156 -> 8.8.8.8 占全部 58 条记录，说明诊断对象高度集中；结合 LOCAL_OUT 上的 DROP 判决，可进一步定位到本机发包规则。

规则过严和攻击防护触发在日志上可能都表现为大量 DROP，因此诊断时需要结合业务背景判断。如果被丢弃的是少数固定目标地址，且触发操作来自用户主动访问，则更可能是规则配置错误或默认策略过严；如果被丢弃的是大量不同源地址访问本机敏感端口，则可能是安全策略正常工作或存在扫描行为。本系统通过唯一源地址数、唯一目的地址数和热门 IP 流统计，为这一判断提供依据。

对于规则过严类故障，系统输出的 skb 生命周期也有参考意义。若同一 skb 只经历 `nf_hook_slow` 后即出现 DROP，说明处理链路很短，规则可能在较早位置直接拒绝；若同一 skb 经历多个函数后才在某个 Hook 被拒绝，则说明故障位置更靠后。测试日志中的平均处理步数较低，符合本地输出阶段被快速拒绝的特征。

### 5.4 入站服务不可达诊断

入站服务不可达的典型场景是外部主机无法访问本机 Web、SSH 或数据库服务。诊断时应按本机服务 IP 或端口相关流量过滤，并重点观察 PRE_ROUTING 与 LOCAL_IN。若 PRE_ROUTING 可见目标为本机的包，但 LOCAL_IN 或 nf_hook_slow 返回 DROP，说明包已经进入主机网络栈但被入站链规则拒绝；若连 PRE_ROUTING 都没有记录，则更可能是上游网络、网卡、路由或过滤条件问题。该流程把“服务不可达”拆成是否进入主机、是否到达入站链、是否被规则拒绝三个判断点。

入站诊断还需要注意 DNAT 和端口转发场景。如果主机承担网关或负载转发功能，外部访问的目标地址可能在 PREROUTING 阶段被 DNAT 改写，然后进入 FORWARD 或 LOCAL_IN。此时单纯按最终内网地址过滤可能漏掉转换前的记录，按外部地址过滤又可能看不到转换后的路径。系统目前主要采集 IP 地址和 Hook 判决，尚未采集端口和 NAT 前后地址变化，论文将其作为后续改进方向。

若系统在 LOCAL_IN 上看到 ACCEPT，但应用仍然不可达，则结论不应继续指向防火墙，而应转向服务监听、端口绑定、应用日志或本机 socket 状态。这个判断体现了诊断系统的边界：它能说明内核防火墙是否放行，不能替代所有应用层故障排查。明确边界可以避免把所有连接失败都归因于防火墙。

### 5.5 转发路径异常诊断

对于网关设备，转发故障不能只看本机 INPUT 或 OUTPUT 链。系统通过 hook_num:2 和 ip_forward 相关记录识别 FORWARD 路径。若流量进入 PRE_ROUTING 后进入 FORWARD，并在该 Hook 返回 DROP，则说明转发链规则阻止了数据包；若进入 FORWARD 后缺少 POST_ROUTING 或 ip_finish_output 记录，可能与路由策略、出接口或后续 Hook 判决有关。skb 地址在这里用于串联同一数据包的轨迹，避免把不同包的片段误判为同一条路径。

转发路径的完整证据通常应包括接收入口、PREROUTING、FORWARD、POST_ROUTING 和最终输出。若只看到 PREROUTING 而没有 FORWARD，可能是目标为本机、路由查找失败或策略路由改变了路径；若看到 FORWARD 但返回 DROP，优先检查 FORWARD 链和默认策略；若看到 FORWARD ACCEPT 但没有 POST_ROUTING，则需要检查路由表、出接口状态、反向路径过滤和内核转发开关。系统通过 Hook 分布和函数调用频率帮助用户判断停在哪一段。

网关场景中还常见 SNAT、MASQUERADE 和 DNAT 配置错误。当前系统没有直接解析 NAT 扩展目标，但仍能通过路径缺失和判决结果发现异常。后续若增加端口、协议、接口和 NAT 前后地址字段，就可以把“路径异常”进一步细分为“转发链拒绝”“NAT 未命中”“路由无出口”等更具体结论。

### 5.6 监控无输出诊断

监控无输出本身也是一种故障。系统将其分为环境类和条件类两种情况。环境类包括 bpftrace 未安装、未以 root 权限运行、内核符号不可见或脚本文件缺失；条件类包括没有触发流量、过滤 IP 写错、Hook 选择不符合流量路径、结果过滤过窄。wrapper 在启动阶段检查 python3、bpftrace 和脚本文件是否存在，用户也可以通过清空过滤条件、执行 ping 8.8.8.8 等方式验证是否有流量进入。

监控无输出的排查应按由外到内的顺序进行。先确认工具能运行，例如 `bpftrace -V`、脚本是否存在、是否具有执行权限、是否使用 root 权限。再确认确实有目标流量，例如执行 `ping` 或 `curl` 触发请求。随后清空过滤条件，观察全量模式是否有输出。若全量模式有输出而过滤模式无输出，说明过滤条件与实际流量不匹配；若全量模式也无输出，则应检查内核符号、函数名和 bpftrace 运行错误。

这类诊断在论文中看似偏工具自身，但对系统可用性非常关键。一个诊断系统如果无法解释“为什么没有数据”，使用者就可能把工具问题误认为网络正常。本文将监控无输出纳入故障模型，使系统从采集、过滤到分析都具备基本自检思路。

### 5.7 数据提取异常诊断

当输出中出现 0.0.0.0、IP 字段缺失或 skb 生命周期不连续时，不能直接得出防火墙拒绝结论。该类问题可能来自 skb 数据指针位置、分片包、特殊封装或探针函数参数位置差异。系统诊断策略是先判断 nf_hook_slow 是否仍能输出 Hook 编号和判决结果，再结合其他 IP 层函数输出复核。如果只有地址字段异常而 Hook 与判决仍存在，结论应写为“采集字段异常，需要改进提取方式”，而不是误报为网络连通性故障。

数据提取异常可以分为参数提取异常和协议头提取异常。参数提取异常通常由函数原型理解错误导致，例如把 `arg2` 当作 `arg0` 使用；协议头提取异常则与 `skb` 内部指针位置有关。本文采集脚本已针对不同函数设置不同参数位置，但仍然存在直接读取 `skb->data` 的局限。诊断报告中保留这一限制，有助于说明系统结论的适用边界。

在改进方案上，可以在采集脚本中增加更多校验，例如检查 IP 版本字段是否为 4、IP 头长度是否合理、源目的地址是否为全零，发现异常时输出专门的 `[PARSE_ERROR]` 记录。分析器再把这类记录纳入“采集异常”统计，而不是丢弃或误解为普通网络记录。这样能够提高系统在复杂网络环境中的可信度。

### 5.8 测试结果分析

表5-1 汇总了典型故障、观测数据和诊断结论之间的映射关系。测试表明，系统能够把原始日志转化为故障阶段和可能原因。对于本机输出受阻样例，58 条记录集中在同一 IP 对，30 次命中 LOCAL_OUT，29 次出现 DROP 相关返回，诊断结论与防火墙输出链规则或本地安全策略相符。

从测试结果可以看出，系统的诊断有效性不依赖单一字段，而依赖字段之间的互相印证。IP 对确定故障对象，Hook 编号确定阶段，DROP 返回确定处理结果，重复记录说明故障具有持续性，skb 地址说明记录来自内核包处理路径。若只看到 DROP 而没有 Hook，无法定位阶段；若只看到 Hook 而没有 DROP，无法说明被拒绝；若只看到 IP 对而没有 skb 和函数记录，则无法证明该流量进入了目标内核路径。

测试也暴露出当前实现中的两个问题。其一，分析器对 `[DROP=-1]` 的归一化不足，导致本地报告中的丢包统计可能低估实际 DROP 数量。其二，当前样例主要验证本机输出受阻，对入站、转发和 NAT 场景缺少更多真实日志支撑。后续测试应通过构造 INPUT DROP、FORWARD DROP、DNAT 配置错误、过滤条件过窄等场景扩展样本，使诊断规则得到更充分验证。

表5-1 故障类型、观测特征与诊断结论

| 故障类型 | 关键观测特征 | 诊断结论 |
|---|---|---|
| 本机输出受阻 | hook_num:3 与 DROP 集中出现 | 检查 OUTPUT/LOCAL_OUT 相关规则 |
| 入站服务不可达 | PRE_ROUTING 可见但 LOCAL_IN DROP | 检查 INPUT/LOCAL_IN 规则 |
| 转发异常 | FORWARD DROP 或缺少后续输出路径 | 检查 FORWARD 链、路由和出接口 |
| 规则过严 | 同一 IP 对持续 DROP | 复核精确匹配规则和默认策略 |
| 监控无输出 | 无函数记录或记录数不足 | 检查权限、依赖、流量和过滤条件 |

## 第6章 总结与展望

### 6.1 工作总结

本文设计并实现了一种基于 eBPF 的 Linux 内核防火墙故障诊断系统。系统以 bpftrace 动态探针采集内核网络路径数据，以 shell 包装脚本提供故障导向的过滤和日志能力，以 Python 分析器完成结构化解析和统计诊断。论文围绕防火墙故障的定位需求，将 IP 地址、Hook 编号、skb 地址、函数名和判决结果组织为诊断证据链，避免了只展示监控数据而不解释故障原因的问题。

本文完成的主要工作包括：梳理 Netfilter Hook 与 IPv4 收发路径之间的关系，确定 `ip_rcv`、`ip_forward`、`ip_local_out`、`nf_hook_slow` 等观测点；实现 bpftrace 采集脚本，输出源地址、目的地址、skb、Hook 和判决；实现交互式包装脚本，支持多条件过滤和日志保存；实现 Python 分析器，对日志进行解析、统计和报告生成；结合已有日志样例完成本机输出受阻故障分析。上述工作共同证明，eBPF 动态跟踪可以为防火墙故障提供比传统日志更接近内核处理过程的证据。

### 6.2 创新点与不足

本文工作的主要特点在于把 Netfilter Hook 路径与 eBPF 动态跟踪结合起来，并以故障诊断为中心设计数据模型。系统能够定位本机输出受阻、入站拒绝、转发链异常、规则过严、监控无输出和字段提取异常等问题。不足之处在于当前实现主要面向 IPv4 和 bpftrace 原型环境，字段提取仍依赖 skb->data，日志解析对返回值格式需要进一步归一化，尚未形成完整的可视化界面和持续运行服务。

与普通抓包或规则计数相比，本文系统的特点在于能够把数据包路径和防火墙判决放在同一条记录链中分析。抓包工具能看到数据包是否出现在网卡或接口上，却无法直接说明 `nf_hook_slow` 返回了什么；规则计数能显示某条规则命中次数，却不一定能关联到具体 skb 生命周期。本文系统通过 eBPF 探针补足这两者之间的空白，使诊断结论更接近内核真实处理过程。

系统不足也比较明确。当前实现缺少端口、协议、接口、进程和网络命名空间字段，导致诊断粒度仍停留在 IP 和 Hook 层面。对于容器网络、多租户主机和复杂 NAT 场景，仅凭源目的 IP 可能不足以定位具体服务或规则。当前日志分析仍以文本解析为主，面对格式变化时鲁棒性有限。论文承认这些限制，并将其作为后续工程化方向。

### 6.3 后续展望

后续工作可以从三个方向展开。技术实现上，可将 bpftrace 原型迁移为 libbpf 或 libbpf-go 程序，使用 ring buffer 输出结构化事件，并通过 BTF 和 CO-RE 提高跨内核版本适配能力。诊断能力上，可引入端口、协议、进程、网络命名空间和规则快照，使结论能够进一步对应到具体服务或规则。产品形态上，可开发 Web 可视化界面，把 Hook 路径、skb 生命周期和故障结论以时间线方式呈现，从而提升系统在运维场景中的可用性。
